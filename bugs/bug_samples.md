# 项目缺陷与修复记录

> 本文件记录开发和验收过程中实际发现的问题。每条记录都包含发现方式、根因、修复和回归证据，便于面试时说明测试思路。

## BUG-001：默认日志选择依赖文件名，可能选错日志

| 字段 | 内容 |
|---|---|
| 缺陷等级 | 严重 |
| 发现方式 | 自动化测试设计时构造“文件名旧但修改时间新”的日志 |
| 测试环境 | Windows 11、Python 3.14 |
| 当前状态 | 已关闭 |
| 对应用例 | FUNC-010 |

**复现步骤**

1. 创建 `z_old.log` 和 `a_new.log`。
2. 将 `a_new.log` 的修改时间设置得更新。
3. 执行默认日志选择逻辑。

**预期结果**：选择修改时间最新的 `a_new.log`。

**实际结果（修复前）**：程序按名称排序，错误选择 `z_old.log`。

**根因**：使用文件名排序代替修改时间比较。

**修复内容**：使用 `path.stat().st_mtime_ns` 比较日志修改时间。

**回归证据**：`test_find_latest_log_returns_newest_file` 自动化测试通过。

## BUG-002：报告中暴露本机绝对路径

| 字段 | 内容 |
|---|---|
| 缺陷等级 | 一般 |
| 发现方式 | 上传前检查示例报告内容 |
| 测试环境 | Windows 11、Python 3.14 |
| 当前状态 | 已关闭 |
| 对应用例 | FUNC-009、FUNC-011、FUNC-012 |

**复现步骤**

1. 执行 `python analyze_log.py --log logs/sample.log`。
2. 打开 `reports/report.md`。

**预期结果**：项目内日志显示为 `logs/sample.log`。

**实际结果（修复前）**：报告写入 `C:\Users\...` 本机绝对路径。

**根因**：分析结果直接保存传入的绝对路径。

**修复内容**：项目目录内的日志在报告中转换为相对路径。

**回归证据**：命令行集成测试验证报告使用 `logs/sample.log`，且未出现本机用户目录。

## BUG-003：测试用例文档与实现描述不一致

| 字段 | 内容 |
|---|---|
| 缺陷等级 | 一般 |
| 发现方式 | README、测试用例和代码一致性审查 |
| 测试环境 | 文档审查 |
| 当前状态 | 已关闭 |
| 对应用例 | FUNC-010 |

**复现步骤**

1. 阅读 `test_cases/test_cases.md` 中 FUNC-010。
2. 阅读 `analyze_log.py` 中 `find_latest_log()`。

**预期结果**：测试用例应描述按修改时间选择最新日志。

**实际结果（修复前）**：用例仍写成“自动选择名称最新的日志”。

**根因**：代码修复后未同步更新测试文档。

**修复内容**：统一 README、测试用例和代码中的最新日志定义。

**回归证据**：上传前执行文档与实现一致性检查。
